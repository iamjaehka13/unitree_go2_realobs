실로봇 상수 보정용 데이터 수집 프로토콜
====================================

작성일: 2026-03-09
목적:
- 현재 `realdata/`에 있는 `stand` + `cmd_x1,y0,z-1` 로그를 출발점으로,
  배터리/열/threshold 상수를 더 설득력 있게 보정하기 위한 추가 수집 프로토콜을 고정한다.
- 이 문서는 "논문/코드 상수 보정용 추가 로그를 무엇을, 어떤 조합으로, 몇 번, 어떤 형식으로 받아야 하는가"
  에만 집중한다.

주의:
- 기존 런북인 `third_party/PAPER_B_REAL_LOG_RUNBOOK_KR.txt`를 대체하지 않는다.
- 이 문서는 "실험 실행법"보다 "수집 조합과 로그 설계"를 더 상세하게 정의한다.


========================================================================
0. 왜 더 뽑아야 하는가
========================================================================
- 지금 있는 로그:
  1) `just_standdata`
  2) `cmd_x1,y0,z-1data`
- 이 두 개만으로도 1차 보정은 가능하다:
  - pack OCV-SOC baseline
  - pack sag/internal resistance 대략값
  - threshold sanity check
- 하지만 아직 분리해서 보기 어려운 것:
  - "열 때문에 성능이 나빠진 것" vs "SOC/전압 때문에 나빠진 것"
  - 저부하/중부하/고부하에 공통으로 맞는 thermal RC
  - low-SOC 근처의 brownout/hard-stop 경계
  - robot-to-robot / battery-to-battery 편차

핵심:
- 아래 4가지 상태를 분리해서 확보하면 가장 좋다.
  1) 차가운 모터 + 높은 SOC
  2) 뜨거운 모터 + 높은 SOC
  3) 차가운 모터 + 낮은 SOC
  4) 뜨거운 모터 + 낮은 SOC

1. 매 run마다 꼭 받아야 하는 것
========================================================================
A. 필수 원본 로그
- rosbag2 `LowState` 원본
- 현재처럼 `/lowstate`가 500Hz 수준으로 저장되면 충분

B. 필수 command 정보
- 지금까지는 폴더명으로 `cmd_x1,y0,z-1`를 해석했지만, 앞으로는 명령을 명시적으로 같이 저장
- 둘 중 하나는 반드시 필요:
  1) 고정 명령이면 `run_info.json`에 `cmd_vx`, `cmd_vy`, `cmd_wz`
  2) 시간 가변 명령이면 `command.csv`

C. 필수 metadata
- `robot_id`(동명이 로봇개id: 0, 강재하 로봇개: 1)
- `battery_id`
- `run_id`
- `protocol_name`
- `ambient_c`
- `surface`
- `payload_kg`
- `start_soc`
- `notes`

D. 권장 동기화 원칙
- raw bag, command, metadata가 같은 run ID를 공유
- timestamp는 단조 증가 기준 하나로 고정
- 최소한 run 시작/종료 시각은 metadata에 기록

2. 실제로 bag 안에 있어야 하는 신호

- 모터 12축:

- 배터리/BMS:
  - `power_v`, `power_a`

  - `foot_force`, `foot_force_est`

추가로 bag 밖에서라도 꼭 같이 기록해야 하는 것:

- estop / mode transition

3. 파일 형식 규칙
========================================================================
권장 디렉토리 구조:

`YYYYMMDD_robot<R>_battery<B>_<protocol>_soc<NN>_rep<k>/`

예시:
`20260310_robotA_battery2_rest_soc90_rep01/`

각 run 디렉토리 안에는:
- `bag/` 또는 rosbag2 폴더 그대로
- `run_info.json`
- `command.csv` 또는 고정 명령이면 생략 가능

`run_info.json` 예시 필드:
- `run_id`
- `robot_id`
- `battery_id`
- `protocol_name`
- `start_soc`
- `ambient_c`
- `surface`
- `payload_kg`
- `cmd_source`
- `cmd_vx`
- `cmd_vy`
- `cmd_wz`
- `preheat_protocol`
- `notes`

`command.csv` 최소 컬럼:
- `t`
- `vx`
- `vy`
- `yaw_rate`

4. 우선순위 A: 지금 바로 가장 필요한 프로토콜
========================================================================
아래는 "지금 상수 보정에 바로 먹히는" 우선순위 높은 수집 세트다.

--------------------------------------------------------------------------
A1. Rest / OCV baseline
--------------------------------------------------------------------------
목적:
- 부하가 거의 없는 상태에서 `Vpack-SOC` baseline 추정
- low-current OCV 선형식 보정

조건:
- 로봇 자세 부하가 거의 없는 상태가 가장 좋음
- 가능하면 walk/turn 없이 정지

권장 조합:
- `rest_soc90`
- `rest_soc50`
- `rest_soc20`
- 가능하면 `rest_soc10~15`

run당 길이:
- 5~10분

반복:
- 각 조건 최소 2회
- 권장 3회

--------------------------------------------------------------------------
A2. Stand hold
--------------------------------------------------------------------------
목적:
- 정지 자세 유지 시 baseline 전류/온도 드리프트
- 완전 rest와 실제 운용 사이의 중간점 확보

권장 조합:
- `stand_soc90`
- `stand_soc50`
- `stand_soc20`

run당 길이:
- 5분

반복:
- 각 조건 최소 2회

--------------------------------------------------------------------------
A3. 직진 정속
--------------------------------------------------------------------------
목적:
- 부하 증가에 따른 전압 sag
- 지속 부하에서 heating trend 확인

권장 명령:
- `x=0.3, y=0.0, z=0.0`
- `x=0.6, y=0.0, z=0.0`
- `x=1.0, y=0.0, z=0.0`

권장 SOC:
- `90`
- `20`

run당 길이:
- 각 3~5분

반복:
- 각 조건 최소 2회

--------------------------------------------------------------------------
A4. 제자리 회전
--------------------------------------------------------------------------
목적:
- yaw load 특성 분리
- 회전이 thermal / current에 주는 영향 확인

권장 명령:
- `x=0.0, y=0.0, z=+1.0`
- `x=0.0, y=0.0, z=-1.0`

권장 SOC:
- `90`
- `20`

run당 길이:
- 각 3~5분

--------------------------------------------------------------------------
A5. 복합 고부하
--------------------------------------------------------------------------
목적:
- worst-case에 가까운 heating + sag 동시 관측
- 현재 보유한 `cmd_x1,y0,z-1`를 더 체계적으로 반복

권장 명령:
- `x=1.0, y=0.0, z=-1.0`
- 가능하면 `x=1.0, y=0.0, z=+1.0`도 추가

권장 SOC:
- `90`
- `20`
- 가능하면 `10~15`

run당 길이:
- 3~5분

--------------------------------------------------------------------------
A6. Burst step
--------------------------------------------------------------------------
목적:
- 순간 가감속에 대한 전압 sag/recovery 확인
- hysteresis, governor margin 검증

권장 패턴:
- `0 -> x=1.0 -> 0` 반복
- `0 -> z=-1.0 -> 0` 반복
- on/off 3~5초 단위 반복

run당 길이:
- 2~3분

--------------------------------------------------------------------------
A7. Heat then cool
--------------------------------------------------------------------------
목적:
- thermal RC 식별
- 가열 구간과 냉각 구간을 한 run 안에서 모두 확보

권장 절차:
1) 고부하 명령 3~5분
2) 바로 stand 또는 rest로 전환
3) cooling 10분 이상

가장 중요:
- 이 run은 꼭 필요하다
- thermal 상수 맞출 때 제일 정보량이 많다

5. 우선순위 B: "뜨거운 모터 + 높은 SOC"를 명시적으로 만드는 프로토콜
========================================================================
질문에서 나온 "모터 온도는 높은데 배터리는 100% 또는 높은 상태" 조합은 유용하다.

왜 필요한가:
- 전압 부족이 아닌 "순수 thermal effect"를 보기 좋음
- 열 때문에 torque/behavior가 바뀌는 구간을 SOC 영향과 분리 가능

권장 방식:
1) 시작 SOC를 `90~95%`로 맞춤
2) 중~고부하 프로파일로 3~5분 preheat
3) 그 상태를 유지한 채 동일 명령 2~3분 기록
4) 이후 cooling 10분 기록

추천 run ID:
- `hot_highsoc_combo`
- `hot_highsoc_turn`
- `hot_highsoc_cooldown`

주의:
- 꼭 100%일 필요는 없음
- `90~95%`면 충분
- 100% 근처는 balancing / 충전 직후 상태가 섞여 해석이 까다로울 수 있음

6. 최소 세트 / 권장 세트 / 이상적 세트
========================================================================
--------------------------------------------------------------------------
최소 세트
--------------------------------------------------------------------------
- `rest_soc90`
- `rest_soc20`
- `stand_soc20`
- `x0.6_soc90`
- `x1.0_soc20`
- `z-1.0_soc20`
- `x1.0_z-1.0_soc20`
- `burst_x_soc20`
- `heat_then_cool_soc90`

용도:
- 배터리 1차 보정
- threshold sanity check
- thermal trend 확인

--------------------------------------------------------------------------
권장 세트
--------------------------------------------------------------------------
- Rest: `soc90/soc50/soc20`
- Stand: `soc90/soc50/soc20`
- Cruise: `x0.3/x0.6/x1.0` each at `soc90/soc20`
- Yaw: `z+1.0/z-1.0` each at `soc90/soc20`
- Combo: `x1.0,z-1.0` at `soc90/soc20`
- Burst: `x burst`, `yaw burst` at `soc20`
- Heat-then-cool: `soc90`, `soc20`

--------------------------------------------------------------------------
이상적 세트
--------------------------------------------------------------------------
- 위 권장 세트를 2~3대 robot에서 반복
- battery pack도 2개 이상 사용
- 가능하면 robot-battery swap 수행

7. 여러 대 robot으로 수집할 때 권장 설계
========================================================================
결론:
- 여러 마리로 따는 것이 확실히 더 좋다

이유:
- `robot 편차`
- `battery 편차`
- `sensor bias 편차`
를 분리할 수 있다

권장 수준:
- 최소: 1대 robot + 여러 SOC/부하 조건
- 권장: 2대 robot + 2개 battery
- 이상적: 3대 robot + 2개 이상 battery + swap

가장 좋은 설계:
1) Robot A + Battery 1
2) Robot B + Battery 2
3) 같은 프로토콜 반복
4) 그 다음 Battery 1/2를 서로 교체
5) 같은 프로토콜 반복

이렇게 하면:
- robot 차이인지
- battery 차이인지
훨씬 분리해서 해석 가능

8. 프로토콜별 수집 목적 요약
========================================================================
`rest`
- OCV-SOC baseline

`stand`
- 자세 유지 전류 / 저부하 thermal baseline

`straight cruise`
- 부하 증가에 따른 sag / heating

`yaw only`
- 회전 부하 특성

`combo high load`
- worst-case thermal + voltage interaction

`burst`
- 동적 sag / recovery / hysteresis

`heat then cool`
- thermal RC 식별 핵심

`hot_highsoc`
- thermal effect만 분리해서 보기 좋은 조합


========================================================================
9. 안전상 stop rule
========================================================================
실험 중 아래 구간에서는 무리해서 더 끌지 않는다.

- `vcell_min <= 3.05V` 근처
- `pack <= 24.5~25.0V` 근처
- 케이스 온도가 장시간 `75C` 이상
- 하드웨어 보호/경고가 먼저 뜨는 경우

원칙:
- threshold calibration이 목적이지, 하드웨어 한계 실험이 목적이 아님
