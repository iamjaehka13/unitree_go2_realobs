Paper B 실행 문서: 현재 코드 기준으로 맞는 버전

핵심 결론
이 프로젝트의 메인 메시지는 "thermal-aware policy"가 아니라 "deployable measurable-only observation contract"이다.
다만 현재 코드 기준에서 바로 제출 가능한 상태는 아니다.
가장 먼저 해야 할 일은 새 실험을 늘리는 것이 아니라, Baseline/ObsOnly/RealObs/Strategic 비교의 공정성을 맞추는 것이다.

지금 코드에서 이미 맞는 것
1. 실험 taxonomy는 이미 존재한다.
- Baseline: locomotion-only observation/reward
- ObsOnly: measurable-only observation + baseline reward
- RealObs: measurable-only observation + measurable proxy reward
- Strategic: privileged observation/reward + governor on
- TempDose: hard thermal stop side ablation

2. RealObs/Baseline은 deployable track으로 어느 정도 정리되어 있다.
- brownout source는 sensor_voltage
- risk_factor는 1.0으로 고정
- RealObs의 temperature metric semantics는 case_proxy

3. 평가 파이프라인은 existing stack을 source of truth로 쓰는 편이 맞다.
- scripts/rsl_rl/evaluate.py 가 fixed-fault, terminal snapshot, governor diagnostics를 이미 지원한다.
- scripts/rsl_rl/run_paper_b_core_eval.sh 가 12-motor fixed-fault sweep과 governor on/off evaluation을 이미 돌린다.
- scripts/real/log_to_replay_csv.py 와 scripts/real/offline_governor_eval_from_log.py 가 real-log 처리의 시작점으로 이미 있다.

지금 문서에서 틀렸던 핵심
1. thermal termination만 confound라고 보면 안 된다.
더 큰 문제는 termination contract 자체가 ladder에서 다르다는 점이다.
현재 Baseline은 motor_stall termination이 없고, ObsOnly/RealObs는 Strategic termination 체계를 상속한다.
이 상태에서는 Baseline -> ObsOnly 를 "관측만 바뀐 비교"라고 말하기 어렵다.

2. 평가 파이프라인은 가능하면 evaluate.py 를 기준으로 정리해야 한다.
fixed-fault sweep, terminal snapshot, governor metrics는 기존 evaluate.py 기반으로 정리하는 편이 맞다.
얇은 wrapper 나 batch runner 는 괜찮지만, evaluate.py 와 다른 제2 paper protocol을 만들면 안 된다.

3. risk_factor는 main ladder에서는 핵심 confound가 아니다.
Baseline/RealObs는 이미 risk_factor를 1.0으로 고정한다.
다만 Strategic 쪽에서는 여전히 privileged command/voltage semantics 묶음의 일부로 남는다.
더 중요한 privileged 차이는 Strategic이 bms_pred 기반 voltage semantics를 쓴다는 점이다.

4. measurable-only claim은 아직 방어력이 약하다.
RealObs helper는 case temperature가 없으면 coil_temp - delta 로 fallback한다.
논문에서 "measurable-only"를 강하게 주장하려면 이 fallback을 문서화하거나 차단해야 한다.

우선순위
Priority 1. Main ladder의 termination contract 정렬
목표:
- Baseline
- ObsOnly
- RealObs
이 세 개는 "정보와 보상"만 다르고 termination은 최대한 같아야 한다.

권장 메인 비교 규약:
- 핵심 원칙: Baseline, ObsOnly, RealObs는 동일 termination contract를 사용
- 실무적으로 유력한 후보: time_out, base_contact, bad_orientation, motor_stall
- thermal_failure: 메인 ladder에서는 off
- hard thermal stop은 TempDose 또는 별도 side ablation으로 분리

즉, 메인 표는 "soft safety under same termination contract"로 가고,
hard thermal stop은 메인 표에 섞지 않는다.
motor_stall을 포함할지 제외할지는 claim에 따라 결정할 수 있지만, 세 환경에서 동일해야 한다.

Priority 2. Strategic를 upper bound로 명시하고 main fair ladder와 분리
Strategic는 다음 차이를 동시에 가진다.
- privileged observation
- privileged reward
- brownout semantics가 bms_pred
- governor on
- thermal hard stop on

따라서 Strategic는 같은 표에 두더라도 "upper bound" 열로 명시해야 한다.
RealObs와 정면 비교하려면 최소한 아래 중 하나가 필요하다.
- Strategic-noGov
- Strategic-SoftTherm

단, 이것도 Priority 1 이후다.

Priority 3. measurable-only claim을 코드 레벨에서 잠그기
해야 할 일:
- RealObs observation/reward에서 bms_pred, coil_temp 직접 경로가 policy-facing channel로 들어가지 않음을 점검
- case-proxy channel이 항상 존재하도록 runtime state/sensor model을 보장하거나
- fallback 사용 시 "simulated measurable proxy"라고 문서에 명확히 적기

가장 안전한 방향:
- RealObs는 motor_case_temp 기반만 사용
- fallback은 assertion 또는 explicit proxy-generation 단계로 한정

Priority 4. 기존 평가 파이프라인을 paper protocol로 고정
새 평가 로직을 따로 발명하지 말고 기존 것을 기준으로 고정한다.

평가 규약:
- fixed-fault sweep: motor 0..11
- num_episodes, num_envs, command profile, governor mode를 명시
- evaluate.py 의 terminal snapshot 사용
- gait/contact/governor diagnostics는 existing CSV output 재사용
- wrapper를 추가하더라도 metric semantics와 fault semantics의 source는 evaluate.py 여야 한다.

최소 제출 세트:
- no-fault eval
- fixed-fault 12-motor sweep
- governor off/on evaluation
- TempDose side ablation

Priority 5. artifact는 마지막에 묶기
artifact bundle은 필요하지만 1순위가 아니다.
실험 protocol이 흔들리는 상태에서 artifact부터 만들면 다시 버리게 된다.

권장 artifact 최소 세트:
- frozen cfg
- checkpoint
- evaluate.py invocation script
- fixed-fault CSV
- summary CSV
- sensor mapping 문서

실제로 추가해야 할 새 것
필수
1. Main ladder termination alignment patch
- 목적은 Baseline, ObsOnly, RealObs가 동일 termination contract를 가지게 만드는 것이다.
- 방법 A: Baseline에 motor_stall을 추가
- 방법 B: ObsOnly/RealObs termination을 Baseline 쪽으로 낮춤
- degraded actuator integrity를 메인 문제로 본다면 방법 A가 더 자연스럽다.

2. Strategic comparison variant 최소 1개
- Strategic-noGov 또는 Strategic-SoftTherm
- 둘 다 만들 수 있으면 더 좋지만 최소 1개는 있어야 policy vs intervention 분리가 된다.

3. measurable-only assertion or audit
- RealObs observation/reward contract를 코드와 문서로 고정

선택
1. RealObs+Gov training variant
주의:
- evaluation override로 "같은 policy + governor on/off"는 이미 볼 수 있다.
- 따라서 RealObs+Gov task를 꼭 새로 만들 필요는 없다.
- "governor 하에서 학습된 정책" 자체를 주장할 때만 추가한다.

2. channel/reward ablation cfg
이건 논문 완성도를 높이지만 최우선은 아니다.
비교 공정성이 먼저다.

논문 메시지 정리
메인 claim:
"우리는 실제 로봇에서 측정 가능한 채널만 허용하는 observation contract 아래서도,
degraded dynamics 환경에서 안전성과 지속성을 개선할 수 있음을 보인다."

메인 ladder:
- Baseline: locomotion only
- ObsOnly: measurable-only observation only
- RealObs: measurable-only observation + measurable proxy safety reward

Upper bound:
- Strategic: privileged + intervention upper bound

Side ablation:
- TempDose: hard thermal stop semantics

메트릭 우선순위
1. tracking
2. survival time
3. min voltage / min cell voltage
4. max temp
5. motor-wise worst-case over 12 motors
6. governor active ratio / latch ratio
7. termination reason fraction

하지 말아야 할 것
1. evaluate.py 를 우회하는 제2 평가 프로토콜을 만들지 말 것
2. synthetic calibration plot을 "real calibration"처럼 쓰지 말 것
3. Strategic를 main deployable result처럼 쓰지 말 것
4. Baseline과 ObsOnly/RealObs의 termination 차이를 무시한 채 ladder를 주장하지 말 것

실행 순서
Step 1. Baseline/ObsOnly/RealObs termination contract 통일
Step 2. evaluate.py 를 source of truth로 fixed-fault protocol 고정
Step 3. Strategic-noGov 또는 Strategic-SoftTherm 추가
Step 4. measurable-only contract audit/assertion 추가
Step 5. 필요한 경우 observation/reward ablation 추가
Step 6. 마지막에 artifact bundle 정리

한 줄 판단
이 프로젝트는 이미 "논문이 될 구조"는 갖췄다.
하지만 지금 중요한 건 새 아이디어를 더 붙이는 게 아니라,
"현재 코드로 진짜 공정한 비교를 하고 있는가"를 먼저 맞추는 것이다.
